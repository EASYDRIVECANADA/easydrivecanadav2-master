# Owini-Style Facebook Browser Assistant Design

## Goal

Add a Phase 2 workflow that helps staff post EasyDrive inventory to Facebook Marketplace with local browser assistance. The feature should replicate the useful part of an Owini-style workflow: take prepared inventory data, open Facebook Marketplace, fill the listing form, and stop before final submission so staff can review and post manually.

This phase must not store Facebook credentials, cookies, two-factor secrets, or run unattended background posting.

## Current Context

Phase 1 already added a Facebook Marketplace posting assistant at `/admin/marketplace/facebook`. It generates copy-ready listing payloads from `edc_vehicles`, tracks posting readiness/status in `edc_facebook_marketplace_posts`, and lets staff copy text and open Facebook manually.

The next step is not a hosted server bot. A Next/Vercel app cannot reliably and safely control a user's Facebook browser session. The right technical shape is a local browser assistant that runs on the dealership computer where the staff member is already logged into Facebook.

## Recommended Approach

Build a local Playwright-based runner that connects to EasyDrive, loads one prepared post payload, opens a visible browser profile, fills Facebook Marketplace vehicle listing fields, and pauses for manual review.

The admin web app remains the source of truth for inventory and posting status. The local runner performs browser interaction only. The final Facebook Post button remains a staff action.

## User Stories

- As a staff member, I can click `Assist Post` from a ready Facebook queue item.
- As a staff member, I can use my already logged-in Facebook browser profile without giving EasyDrive my password.
- As a staff member, I can watch Facebook Marketplace open and have the listing fields filled from EasyDrive data.
- As a staff member, I can review the form before posting.
- As a staff member, I can return to EasyDrive and mark the item posted or paste the Facebook listing URL.
- As an admin, I can see whether browser assistance was started, completed to review, failed, or posted.

## Phase 2 Scope

### Admin UI

Extend `/admin/marketplace/facebook` with a new action on ready vehicles:

- `Assist Post`
- Disabled when readiness is not ready.
- Shows a small setup hint if the local runner is not available.
- Shows the generated payload before launching assistance.

Add status visibility:

- `assist_ready`
- `assist_started`
- `needs_review`
- `assist_failed`

These are operational assistance states, not replacements for the existing posting statuses. The durable posting status remains `draft`, `ready`, `posted`, `needs_update`, `sold_remove`, `skipped`, or `failed`.

### Local Runner

Create a local Node/Playwright runner that staff can start from the dealership computer.

Responsibilities:

- Open a visible Chromium/Edge browser with a persistent local profile.
- Reuse the staff member's existing Facebook login if present.
- Navigate to Facebook Marketplace vehicle listing creation.
- Fill supported fields from the EasyDrive post payload.
- Upload or attach images only when feasible from accessible URLs/files.
- Pause before final submit.
- Report run status back to EasyDrive.

The runner should never:

- Store Facebook username or password.
- Store Facebook cookies in Supabase.
- Solve CAPTCHA.
- Bypass two-factor authentication.
- Click the final Facebook `Post` button.
- Run headless mass posting.

### EasyDrive API

Add a runner-safe API surface:

- `GET /api/admin/marketplace/facebook/posts/[id]/assist`
  - Returns the prepared payload for one post row.
  - Includes vehicle title, description, price, mileage, location, VIN, stock, image URLs, and public URL.
  - Requires admin authorization.
- `PATCH /api/admin/marketplace/facebook/posts/[id]/assist`
  - Records assist status, error message, and timestamps.
  - Does not mark `posted` automatically.

The runner can authenticate using a short-lived token generated by the admin session or a manually copied one-time launch token. Do not use service-role keys in the local runner.

### Launch Model

Use a conservative launch model:

1. Staff starts the local runner from the dealership computer.
2. Runner shows a local URL or status line: `Ready for EasyDrive assist requests`.
3. Staff clicks `Assist Post` in EasyDrive.
4. EasyDrive either:
   - opens a custom local URL such as `http://127.0.0.1:<port>/assist?token=...`, or
   - copies a command/token for the runner.
5. Runner fetches the payload from EasyDrive.
6. Runner opens Facebook and fills the listing.
7. Runner sets status to `needs_review`.
8. Staff reviews and manually posts.
9. Staff marks the row `posted` and saves the Facebook listing URL in EasyDrive.

The custom local URL is the preferred UX if browser security and deployment constraints allow it. The command/token fallback is acceptable for the first internal version.

## Data Model

The existing `edc_facebook_marketplace_posts` table can support Phase 2 with minimal additions.

Add columns in a new migration:

- `assist_status text`
- `assist_started_at timestamptz`
- `assist_completed_at timestamptz`
- `assist_error text`
- `assist_payload jsonb not null default '{}'::jsonb`

Allowed `assist_status` values:

- `not_started`
- `started`
- `needs_review`
- `failed`
- `cancelled`

Keep the existing `status` column for durable listing status. Assistance can succeed even if staff later chooses not to post.

## Field Mapping

The first assisted version should try to fill only stable fields:

- title
- price
- description
- mileage
- location
- VIN when Facebook exposes a compatible field
- vehicle category/type when obvious

Images are more fragile. The runner should attempt image upload only if it can reliably download the prepared image URLs to temporary local files. If Facebook blocks or changes image upload behavior, the runner should stop with a clear message and leave staff with the copy-ready Phase 1 workflow.

## Error Handling

- Runner not running: admin UI shows setup hint.
- Facebook not logged in: runner opens Facebook and waits for staff login.
- CAPTCHA or two-factor prompt: runner pauses and asks staff to complete it manually.
- Facebook form changed: runner records `assist_failed` with the field that failed.
- Image upload failed: runner can continue with text fields and warn staff.
- Staff cancels: runner records `cancelled`.
- EasyDrive API unavailable: runner leaves the browser untouched and shows a local error.

## Security And Risk Controls

- No Facebook credentials in EasyDrive.
- No Facebook cookies in Supabase.
- No background posting loop.
- No final submit click.
- Visible browser only.
- One post per explicit staff action.
- Local runner token must be short lived.
- All assist status updates must be auditable in EasyDrive.

This design lowers account risk but does not eliminate it. Facebook Marketplace may still detect automation or change the form. The client must understand that the feature is an assistant, not an official Facebook integration.

## Testing

Unit tests:

- Assist status validation.
- Assist payload generation from existing Facebook post rows.
- One-time token validation helper.
- Runner command/url generation.

API tests:

- Assist payload route requires authorization.
- Assist payload route returns prepared listing data.
- Assist status update rejects invalid statuses.
- Assist status update does not mark a post as `posted`.

Runner dry-run tests:

- Runner can load a local fixture payload.
- Runner can start and expose a local health endpoint.
- Runner can report status back to a mocked EasyDrive endpoint.
- Runner refuses to run without a launch token.

Manual verification:

- Start runner locally.
- Click `Assist Post` for a ready vehicle.
- Confirm visible browser opens Facebook Marketplace.
- Confirm fields are filled.
- Confirm runner pauses before final submit.
- Confirm EasyDrive row becomes `needs_review`.
- Manually post and save Facebook listing URL.

## Non-Goals

- Fully automated Facebook posting.
- Scheduled reposting.
- Sold-listing removal automation.
- Messenger scraping.
- Lead scraping from Facebook.
- Facebook credential vault.
- CAPTCHA or two-factor bypass.
- Running the automation from Vercel or any hosted server.

## Recommended Build Order

1. Add assist schema fields and helper tests.
2. Add assist API payload/status routes.
3. Add `Assist Post` button and assist status UI.
4. Add local runner scaffold with health endpoint and launch token handling.
5. Add dry-run mode that prints the prepared payload.
6. Add visible browser launch and Facebook navigation.
7. Add field filling one field group at a time.
8. Add status reporting back to EasyDrive.
9. Run manual dealership-computer verification.

## Acceptance Criteria

- Staff can start a local runner without entering Facebook credentials into EasyDrive.
- Staff can launch browser assistance from a ready queue item.
- Browser assistance opens a visible Facebook Marketplace create flow.
- Supported fields are filled from the EasyDrive payload.
- Automation stops before final submit.
- EasyDrive records `started`, `needs_review`, `failed`, or `cancelled`.
- Staff can still use the Phase 1 manual copy workflow if assistance fails.
